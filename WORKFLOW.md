# 작업 가이드 — 레거시 JSP 옷입히기

> 다른 회사(퍼블리싱)가 만든 정적 디자인(HTML+CSS)을 받아, 기존 레거시 JSP에
> **동적 바인딩을 살린 채 입히는** 작업. 신규개발/대규모분석 아님.

---

## 작업 시작 전 (원칙 — 강제)

- **로컬 완전체 구동이 하드게이트.** 앱이 로컬에 안 뜨면 작업 시작 금지. (상상코딩 E2E 차단)
- **우리가 다 돌려보고 넘긴다.** 검증 안 한 결과 핸드오프 금지.
- **뷰레이어만 건드린다.** 백엔드(컨트롤러/서비스/DAO/DB) 로직 수정 금지 — 오류 발견 시 **리포트만**.
- **master 직접 작업 금지.** 항상 feature 브랜치.
- **운영배포 안 한다.** 출하 = 브랜치 커밋/푸시까지. 운영반영·E2E는 우리 몫 아님(세종팀).
- **재작업은 항상 원본에서.** 이미 입힌 JSP에 덧입히지 말 것 — 납품이 수정돼 다시 오므로(원본 스냅샷 = 진실원천). 재skin은 원본에서 새로.

---

## 도구셋

| 슬롯 | 도구 |
|------|------|
| 레퍼런스 | **context7 MCP** — 작업 중 JSTL/EL/태그 문법 실시간 검증 |
| diff 검토 | **`/review`** — 바인딩 보존·목업 잔존 확인 |
| UI 검증 | **`/qa`**(뷰 버그 고침/백엔드 리포트) + **`/browse`**(레이아웃·반응형) |
| 결과 리포트 | **WBS 미러**(내부용 MD, 아래 양식) |
| 출하 | **`git` 브랜치 커밋/푸시** |

---

## 작업 시퀀스

### 작업 시작 시 (1회)

```
R. 구조 정찰        recon-request-template.md 던지기(read-only) → 정찰 리포트 산출
                   = 보존필수 태그 / chrome 선처리 대상 / charset 대응 / 페이지별 리스크
                   (소스만 있으면 됨, 앱 안 떠도 가능)
E. 로컬 완전체 기동  하드게이트. 앱이 떠야 이후 검증 진행
C. 공통 chrome 1회  정찰의 'chrome 선처리 대상'(헤더/푸터/메뉴)을 먼저 입힘 → 사람검토
                   (페이지마다 건드리면 충돌 → 반드시 루프 전에 한 번)
```

### 페이지 루프 (반복)

```
1.  대상 JSP + 매핑된 납품 HTML 열기
1.5 바인딩 인벤토리  이 페이지에서 보존할 동적요소 목록화:
                   EL / JSTL / 스크립틀릿 변수 / 커스텀태그 / include / form action
                   (정찰의 '보존필수 태그' + 페이지 리스크 참조)
2.  context7        ← 막힐 때만. JSTL 버전별 문법 다름, 정확본 확인
3.  grafting        정적 마크업 → 납품 마크업 교체 + 1.5의 바인딩 재주입 (계약 준수)
4.  로컬 반영 → 빌드 OK?   NO → 3으로
5.  /review         1.5 인벤토리를 기준으로 보존 검증 (fail-fast: 바인딩 빠짐/목업 잔존)
6.  /qa + /browse   렌더·시각일치·반응형·동적데이터 (아래 '검증 기준')
7.  WBS 리포트 갱신  그 페이지 레코드 채움
8.  분기            clean → 브랜치 커밋  |  flagged → 보류·리포트
```

> **정찰(R)이 모든 후속의 입력이다.** 1.5는 정찰의 보존필수 태그를, C는 chrome 선처리 대상을,
> grafting 계약은 charset 대응을 정찰 리포트에서 받아 쓴다. 정찰 없이 루프 돌리면 step5 검증 기준도 없다.

---

## grafting 계약 (3번 단계 강제 규칙)

1. **하드코딩 금지** — 납품 HTML의 고정값을 그대로 박지 말 것.
2. **목업 데이터 금지** — 샘플 행/더미 텍스트는 원래 동적 바인딩(`<c:forEach>`, `${...}`)으로 되돌릴 것.
3. **변수 형태 유지** — 기존 EL/JSTL/폼 action/include를 새 마크업 위에 그대로 보존.
4. **charset 일치** — 기존 JSP(EUC-KR 가능성) vs 납품 HTML(UTF-8 통상). **한글 깨짐 1순위 지뢰** — 인코딩 맞출 것.

> grafting은 짜면서 context7로 문법 검증(끝나고가 아님).

---

## 검증 기준

**`/review` (정적, 먼저):** 바인딩 하나도 안 빠졌나 · 목업/하드코딩 잔존 없나.

**`/qa` + `/browse` (동적):** 아래 4개 pass면 통과.
```
① JSP/빌드 에러 없이 렌더
② 레이아웃·스타일이 납품과 일치   ★데이터 값은 무시★ (실데이터 vs 목업이라 다른 게 정상)
③ 동적영역이 실데이터로 채워짐   (루프가 샘플 1줄 아니라 N줄)
④ 폼/링크 안 깨짐 + 콘솔에러 없음(납품 JS vs 기존 JS 충돌 체크)
+ 뷰포트 크기별 반응형 동작 확인
```

> `/qa`는 **뷰레이어 버그만 고침.** 백엔드 오류는 리포트만(거의 안 만남).

---

## 분기 규칙

```
clean   = /review pass AND /qa pass(①~④)   → feature 브랜치 커밋
flagged = 둘 중 하나라도 fail               → 보류, 리포트에 사유 기록
                                              사람이 수정 → 5번(/review)부터 재진입
```

> flagged 몇 개 때문에 clean 페이지를 묶어두지 말 것 — clean은 개별로 흘려보냄.

---

## 결과 리포트 (WBS 미러 · 내부용)

검증 결과를 취합한 것 — 새 문서작업이 아님. WBS 진척판으로 그대로 사용.

```
| WBS키 | 페이지(JSP←납품)  | 적용 | UI테스트 | UI검증 | 상태       |
| 2.1   | userList.jsp←p01 | 적용 | pass     | 일치   | 완료       |
| 2.2   | userView.jsp←p02 | 부분 | pass     | 차이有 | 보류(사람) |

배치요약: 15장 · 완료 12 · 보류 2 · 진행 1 · 진척률 80%
```

- 상태값 = WBS 표준: **착수전 / 진행중 / 완료 / 보류**
- `clean→완료`, `flagged→보류(사람검토)`로 자동 대응
- 배치요약 진척률 = PM 보고용

## 울트라코드 (dynamic workflow) — 두 자리

수동이 기본. 울트라코드는 두 군데서 쓴다 — 정당성이 다르다.

**① 구조 정찰 (Phase R) — 주력. read-only, 1회, 거의 항상 제값.**
```
recon-request-template.md → 4렌즈 병렬(taglib/chrome/패턴/인벤토리) → 정찰 리포트 합성
실행형: workflows/recon-legacy-jsp.js  (템플릿에서 파생)
```
- read-only라 충돌 없음. 1회 비용을 전 페이지가 나눠 amortize → 팬아웃 경제학에 딱.

**② grafting 팬아웃 — 보조. 물량 터질 때만.**
```
(정찰 끝난 상태에서) chrome 1회 → body 페이지별 팬아웃(위 루프) → /qa → 리포트 취합 → 푸시
```
- 위 수동 루프를 **그대로 팬아웃으로 감싼 것** — 수동이 견고하면 자동은 거의 공짜.
- 단, **공통 chrome 선처리 + 원본 기준 재작업**이 돼 있어야 충돌/이중적용 안 남.
- 끝에 **사람 게이트**: triage 리포트(clean/flagged) 보고 사람이 출하 결정.

> ≤15페이지면 ②는 본전 애매 — 수동으로도 됨. ①(정찰)은 규모와 무관하게 가치 있음.

## [요청]

레거시 JSP 웹앱을 **READ-ONLY**로 구조 정찰해줘. 파일 수정/생성/삭제 금지, 읽기만.
목적: 납품된 정적 HTML+CSS를 이 JSP들에 입히되 **동적 바인딩을 보존**하는 grafting 작업의 사전이해.
"마크업 갈아끼우기 전에 반드시 이해/보존해야 할 것"에 초점. 경로는 구체적으로 인용하고, 모르면 "불명"으로.

다음 4가지를 분석해줘:

### ① taglib / TLD 카탈로그
- `web.xml`, `*.tld`, JSP의 `<%@ taglib %>` 선언을 전수 조사.
- 각 커스텀 태그가 **동적 출력(보존 필수)**인지 **정적 래퍼**인지 판별 — TLD/태그핸들러까지 추적해서.
- 표준 JSTL(`c:`,`fmt:`,`fn:`)은 prefix만 정리.
- **왜:** "이 태그는 데이터를 뱉으니 절대 정적치환 금지"를 grafting이 알아야 하니까.

### ② include / chrome 트리
- `<%@ include %>`, `<jsp:include>`, Tiles/SiteMesh/decorator 사용 여부 조사.
- 공유 chrome(헤더/푸터/메뉴/레이아웃)을 식별 → **"한 번만 입히면 상속되는 것" vs "페이지별 body"** 분리.
- **왜:** grafting을 "공유 chrome 1회 선처리 → 페이지 body 개별"로 나누는 근거. 공유파일을 페이지마다 건드리면 충돌남.

### ③ 스크립틀릿 / 바인딩 패턴
- `<% ... %>` 사용 강도 + 반복 관용구 수집: 권한별 조건부 마크업, 페이징, 폼 post, 목록 루프(`c:forEach`), 속성 안 EL(`value="${..}"`) 등.
- 각 관용구마다 **"grafting 시 보존 규칙"** 한 줄.
- **왜:** 공통 패턴을 한 번 이해해 전 페이지에 적용 → 같은 실수 반복 방지.

### ④ 페이지 인벤토리 + charset / 버전
- 대상 JSP 목록화. 각 페이지: **charset(EUC-KR/UTF-8)**, 스크립틀릿 유무, 커스텀태그 유무, include 목록, grafting 난이도(low/medium/high).
- 프로젝트 전체 charset, JSTL/Java/Servlet-JSP 버전을 `web.xml`·`pom.xml`·`build.gradle`·JSP 헤더에서 탐지.
- **왜:** charset 한글깨짐 지뢰 사전탐지 + 페이지별 난이도 분류(작업 배분 근거).

## [산출물]

1. **통합 정찰 리포트** (사람이 읽는 마크다운, 4렌즈 통합)
2. **결정사항**
   - 공유 chrome 선처리 대상 파일 + 방침
   - charset 대응 (변환 필요 여부/방향)
   - **보존필수 동적 태그 목록** (정적치환 절대 금지)
3. **페이지별 리스크** (난이도 + grafting 시 조심할 점)
4. **사람 확인 필요 항목** (소스만으론 못 정한 것 — 매핑/타겟브라우저 등)

## [실행 형태]

- **기본(수동):** 이 요청을 그대로 던져 순차 분석. 소규모면 충분.
- **물량 클 때(ultracode):** ①~④를 병렬 렌즈로 팬아웃 → 합성. read-only라 충돌 없음.
